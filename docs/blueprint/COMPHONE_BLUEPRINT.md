# COMPHONE SUPER APP V5.5 Blueprint ฉบับสมบูรณ์

## 1. บทนำและบริบทธุรกิจ

เอกสารฉบับนี้จัดทำขึ้นเพื่อสรุปภาพรวมเชิงสถาปัตยกรรม เชิงกระบวนการ และเชิงฟีเจอร์ของ **COMPHONE SUPER APP V5.5** สำหรับ **ห้างหุ้นส่วนจำกัด คอมโฟนแอนด์อิเลคโทรนิคส์** ซึ่งเป็นธุรกิจที่ดำเนินงานด้านการจำหน่ายคอมพิวเตอร์ โทรศัพท์มือถือ อุปกรณ์ต่อพ่วง อุปกรณ์สำนักงาน งานซ่อมอุปกรณ์ไอที งานติดตั้งกล้องวงจรปิด และงานจัดซื้อจัดจ้างภาครัฐในจังหวัดร้อยเอ็ด โดยมีฐานปฏิบัติการหลักอยู่ที่อำเภอโพนทอง จังหวัดร้อยเอ็ด

ระบบ V5.5 เป็นการต่อยอดจากระบบเดิมที่พัฒนาบน **Google Apps Script + Google Sheets + Google Drive** และเชื่อมต่อกับ **LINE** และ **Telegram** เพื่อยกระดับจากเครื่องมือช่วยงานเฉพาะส่วน ไปสู่ **Super App สำหรับธุรกิจไอทีและงานติดตั้ง** ที่ขับเคลื่อนด้วยข้อมูลจริง การทำงานอัตโนมัติ และ AI โดยมีวิสัยทัศน์สำคัญ 3 ประการ คือ **Zero-Data Entry**, **Cross-Room Automation**, และ **Operational Intelligence**

> **Zero-Data Entry** หมายถึงการลดภาระการพิมพ์และกรอกข้อมูลของช่างให้เหลือน้อยที่สุด โดยให้ข้อมูลเกิดจากการสแกน QR, การส่งรูป, การส่งเสียง, ตำแหน่ง, และการประมวลผลอัตโนมัติของระบบ

> **Cross-Room Automation** หมายถึงการทำให้ข้อมูลไหลข้ามห้องแชทและข้ามแผนกโดยอัตโนมัติ ตั้งแต่ฝ่ายขาย ช่าง การเงิน ไปจนถึงจัดซื้อ โดยไม่ต้องคอยตามงานแบบแมนนวล

## 2. เป้าหมายเชิงธุรกิจและผลลัพธ์ที่ต้องการ

COMPHONE SUPER APP V5.5 ถูกออกแบบมาเพื่อแก้ปัญหาเชิงปฏิบัติการจริงของธุรกิจ ได้แก่ งานตกหล่น การอัปเดตสถานะไม่ทัน การจัดเก็บรูปหน้างานไม่เป็นระบบ การตัดสต๊อกไม่ครบ การเก็บเงินล่าช้า การติดตามลูกค้าเก่าไม่ต่อเนื่อง และข้อจำกัดของข้อมูลกระจัดกระจายอยู่หลายช่องทาง

| เป้าหมายหลัก | คำอธิบาย | ผลลัพธ์ที่ต้องการ |
| --- | --- | --- |
| **ลดภาระงานเอกสารหน้างาน** | ให้ช่างใช้การสแกน การส่งรูป และการส่งเสียงแทนการพิมพ์ | สถานะงานและบันทึกหน้างานถูกต้องขึ้น ใช้เวลาน้อยลง |
| **เพิ่มความโปร่งใสของงาน** | ทุกกิจกรรมสำคัญถูกบันทึกลงฐานข้อมูลกลางแบบใกล้เคียงเวลาจริง | ผู้บริหารติดตามงานได้ทันที ลดการตามงานซ้ำ |
| **ควบคุมคุณภาพงานติดตั้ง** | ใช้ AI วิเคราะห์ภาพ Before/After และความเรียบร้อยหน้างาน | งานมีมาตรฐานสูงขึ้น ลดการแก้งานย้อนหลัง |
| **ยกระดับการเงินและการขาย** | สร้างใบเสนอราคา รับเงิน ตรวจสลิป และออกเอกสารให้เร็วขึ้น | ปิดการขายไวขึ้น รับเงินเร็วขึ้น ลดงานเอกสาร |
| **บริหารสต๊อกและเครื่องมืออย่างแม่นยำ** | รู้ว่าของอยู่ที่ใคร อยู่รถคันไหน และอะไรใกล้หมด | ลดของหาย ลดการขาดของหน้างาน |
| **ต่อยอดรายได้จากลูกค้าเดิม** | ใช้ข้อมูลติดตั้งและประวัติบริการทำ CRM และงานการตลาด | เกิดรายได้ซ้ำจากงาน PM และบริการหลังการขาย |

## 3. หลักการออกแบบของระบบ

จากข้อมูลโครงการและเอกสารประกอบ ระบบ V5.5 ควรยึดหลักการออกแบบดังต่อไปนี้อย่างเคร่งครัด

### 3.1 Operational Principles

1. **ใช้ข้อมูลจริงเป็นฐาน** ทุกการอัปเดตต้องอิงจากฐานข้อมูล ระบบจริง หรือ API จริง ไม่สร้างข้อมูลสมมติแทนข้อเท็จจริง
2. **ไม่ทำงานแบบเงียบ** หากเกิดความผิดพลาด ระบบต้องบันทึกและแจ้งเหตุให้ตรวจสอบได้
3. **ทำงานต่อเนื่องจนจบงาน** หลีกเลี่ยง flow ที่ทำให้กระบวนการค้างกลางทางโดยไม่มีสถานะชัดเจน
4. **ความถูกต้องสำคัญกว่าความสวยงาม** โดยเฉพาะข้อมูลการเงิน สต๊อก และสถานะงาน
5. **รองรับการขยายตัวแบบค่อยเป็นค่อยไป** โดยยังใช้ประโยชน์จาก Google Workspace ecosystem เดิมให้เต็มที่

### 3.2 COMPHONE CORE RULES ที่ต้องสะท้อนในระบบ

แก่นของระบบ V5.5 ต้องสอดคล้องกับกฎหลักของโครงการ เช่น การไม่โกหกข้อมูล การแจ้ง error ทุกครั้ง การห้าม silent failure การตอบ/แสดงผลตามสถานะจริง การใช้ smart memory อย่างประหยัด และการเน้นความถูกต้องเหนือความดูฉลาด กฎทั้ง 23 ข้อในเอกสารต้นทางจึงควรถูกแปลงเป็นแนวปฏิบัติของระบบ เช่น **error logging**, **status tracking**, **idempotent processing**, **chunked processing**, และ **fallback handling** ในระดับสถาปัตยกรรม

## 4. สถาปัตยกรรมระบบโดยรวม

### 4.1 สถาปัตยกรรมหลัก

ระบบ V5.5 ยังคงใช้โครงสร้างที่เหมาะกับธุรกิจขนาดกลางและการปรับใช้เร็ว ได้แก่ Google Apps Script เป็น backend logic, Google Sheets เป็นฐานข้อมูลเชิงปฏิบัติการ, Google Drive เป็น storage สำหรับไฟล์, และ LINE/Telegram เป็นช่องทางสื่อสารหลัก โดยเสริม API ภายนอกสำหรับงาน AI, QR รับเงิน, การตรวจสลิป, และบริการแผนที่

| ชั้นระบบ | องค์ประกอบหลัก | หน้าที่ |
| --- | --- | --- |
| **ช่องทางรับ-ส่งข้อมูล** | LINE OA, LINE Messaging API, LINE LIFF, Telegram Bot | รับข้อความ รูป เสียง Location และแสดง UI บางส่วน |
| **ชั้นประมวลผลธุรกิจ** | Google Apps Script | ทำหน้าที่ webhook receiver, router, workflow engine, scheduler, document generator |
| **ชั้นข้อมูล** | Google Sheets | เก็บข้อมูลใบงาน ลูกค้า ช่าง สต๊อก บิล คิวรูปภาพ และ log |
| **ชั้นไฟล์และหลักฐาน** | Google Drive | เก็บรูป Before/After, สลิป, เอกสาร PDF, ไฟล์ audit |
| **ชั้น AI และ API ภายนอก** | OpenAI / Gemini, Slip Verification API, QR API, Google Maps API | ถอดเสียง วิเคราะห์รูป สร้างคอนเทนต์ ตรวจสลิป ทำ geofencing และ heatmap |

### 4.2 Data Flow หลัก

1. ช่างหรือพนักงานส่งข้อมูลเข้าระบบผ่าน LINE หรือ Telegram
2. ระบบรับ webhook เข้าสู่ Google Apps Script
3. Script ทำ routing และ business logic ตามชนิดข้อมูล เช่น ข้อความ รูป เสียง หรือ location
4. หากเป็นงานที่ต้องใช้ AI หรือบริการภายนอก ระบบจะเรียก API ที่เกี่ยวข้อง
5. ผลลัพธ์ถูกบันทึกลง Google Sheets และไฟล์ที่เกี่ยวข้องถูกเก็บใน Google Drive
6. ระบบส่งผลลัพธ์กลับเป็นข้อความแจ้งเตือน เอกสาร PDF QR รับเงิน หรือคำสั่งข้ามห้องแชท

### 4.3 ทิศทางสถาปัตยกรรมเชิงปฏิบัติการ

Blueprint นี้เสนอให้ระบบใช้แนวทาง **event-driven workflow แบบ lightweight** บน Google Apps Script กล่าวคือแต่ละเหตุการณ์สำคัญ เช่น เปิดงาน, เปลี่ยนสถานะ, รับรูป, ตรวจสลิป, ปิดงาน, สินค้าต่ำกว่าจุดสั่งซื้อ จะทำหน้าที่เป็น trigger ให้โมดูลถัดไปทำงานต่อโดยอัตโนมัติ พร้อมบันทึก log และสถานะทุกครั้ง เพื่อให้สามารถตรวจสอบย้อนหลังได้ง่าย

## 5. ฟีเจอร์หลักทั้งหมดของ COMPHONE SUPER APP V5.5

เอกสารต้นทางแบ่งฟีเจอร์ออกเป็น 4 กลุ่มหลัก และจากการออกแบบเพิ่มเติมได้ขยายขอบเขตไปสู่ฟีเจอร์ใหม่ที่ช่วยให้ระบบครบวงจรยิ่งขึ้น รายละเอียดรวมทั้งหมดมีดังนี้

### 5.1 กลุ่มที่ 1: หน้างานและการติดตั้ง (Field Operations)

กลุ่มนี้เป็นหัวใจของแนวคิด **Zero-Data Entry** และเป็นพื้นที่ที่ระบบต้องทำให้ช่างทำงานได้เร็วที่สุด แต่ยังเก็บหลักฐานครบที่สุด

#### 5.1.1 ระบบสแกน QR Code 12 สถานะ (Job State Machine)

ระบบนี้ทำหน้าที่เป็นแกนกลางของการติดตามวงจรชีวิตใบงาน โดยให้ช่างเปลี่ยนสถานะผ่านการสแกน QR หรือปุ่มใน LIFF แทนการพิมพ์ข้อความยาว ๆ ลงแชท ระบบต้องบันทึกเวลา ผู้ปฏิบัติงาน และตำแหน่งได้ทุกครั้งที่มีการเปลี่ยนสถานะ

| องค์ประกอบ | รายละเอียดที่ต้องรองรับ |
| --- | --- |
| **ตัวระบุงาน** | Job ID, QR ประจำงาน, ผู้รับผิดชอบ |
| **การอัปเดตสถานะ** | บันทึก Timestamp, User ID, สถานที่, หมายเหตุ |
| **การติดตามย้อนหลัง** | เก็บ timeline ลงตาราง log แยกจากตารางใบงานหลัก |
| **การแจ้งเตือนอัตโนมัติ** | เปลี่ยนสถานะแล้วต้องสามารถ trigger ข้ามห้องได้ |

เพื่อให้รองรับความต้องการของ V5.5 อย่างครบวงจร แนะนำให้ใช้ชุดสถานะอ้างอิง 12 ขั้นดังนี้ โดยสามารถปรับชื่อเรียกได้ตาม workflow จริง แต่ควรคงจำนวนและลำดับสถานะไว้เพื่อการวิเคราะห์ข้อมูลในอนาคต

| Status Code | สถานะอ้างอิง | ความหมายเชิงปฏิบัติการ |
| --- | --- | --- |
| 1 | เปิดงาน | ระบบสร้างใบงานแล้ว |
| 2 | มอบหมายงาน | ระบุช่างหรือทีมรับผิดชอบแล้ว |
| 3 | เตรียมอุปกรณ์ | อยู่ระหว่างเตรียมของ/ตรวจความพร้อม |
| 4 | เดินทาง | ทีมช่างออกเดินทางแล้ว |
| 5 | ถึงหน้างาน | ถึงพิกัดลูกค้าแล้ว |
| 6 | เริ่มปฏิบัติงาน | เริ่มติดตั้ง/ซ่อม |
| 7 | รอลูกค้า/รออนุมัติ | หน้างานค้างเพราะรอการตัดสินใจ |
| 8 | รออะไหล่/รอของ | งานสะดุดเพราะอุปกรณ์ไม่พร้อม |
| 9 | ดำเนินงานต่อ/แก้ไข | กลับเข้าทำงานอีกครั้งหลังค้างงาน |
| 10 | ติดตั้งเสร็จ | งานหน้างานเสร็จแล้ว รอปิดเอกสาร |
| 11 | รอชำระเงิน/ส่งมอบ | รอขั้นตอนการเงินหรือการยืนยันจากลูกค้า |
| 12 | ปิดงานสมบูรณ์ | งาน เอกสาร และสถานะการเงินครบ |

#### 5.1.2 ระบบคัดแยกรูปภาพอัจฉริยะและ Vision Agent (Smart Photo Handling & Vision Agent)

นี่คือหนึ่งในฟีเจอร์ที่ทรงพลังที่สุดของ V5.5 เพราะรวมทั้งการรับรูปจากหลายช่องทาง การจัดเก็บอัตโนมัติ การใช้ AI วิเคราะห์คุณภาพงาน และการต่อยอดสู่การตลาดเข้าไว้ใน workflow เดียว

| ขั้นตอน | รายละเอียดเชิงลึก |
| --- | --- |
| **1. Omnichannel Ingestion** | รับรูปจาก LINE Group, Telegram Bot และ URL ภายนอก แล้วดึงเข้าสู่ระบบกลางเดียวกัน |
| **2. Auto-Routing & Drive Sorting** | จัดเก็บรูปลง Google Drive ตามลูกค้า/Job ID/เฟสของงานโดยอัตโนมัติ ลดความรกในห้องแชท |
| **3. Vision Field Agent** | ใช้ AI แยก Before/After, วิเคราะห์ความเรียบร้อยหน้างาน, ตรวจขยะ/สายไฟ/มุมกล้อง, และใช้ GPS/EXIF ทำ geofencing |
| **4. AI Marketing Content** | เมื่อมีรูป After และงานจบ ระบบคัดรูปดี สร้าง collage และเขียนแคปชั่นรีวิวเพื่อส่งให้อนุมัติเป็นร่างโพสต์ |

องค์ประกอบนี้ควรถูกออกแบบให้เชื่อมต่อโดยตรงกับตาราง `DB_PHOTO_QUEUE` ซึ่งจากการตรวจสอบพบว่าโครงสร้างปัจจุบันมีความพร้อมสูงมากและรองรับข้อมูลเชิง AI อยู่แล้ว เช่น `AILabel`, `AIPhase`, `AIIssues`, และ `ProcessedTimestamp`

#### 5.1.3 Geofencing และการยืนยันหน้างานจริง

แม้ในเอกสารหลักฟังก์ชันนี้จะถูกรวมไว้ใน Smart Photo Handling แต่ในเชิงสถาปัตยกรรมควรถือเป็นกลไกควบคุมความถูกต้องของหน้างานโดยเฉพาะ ระบบต้องใช้ข้อมูลพิกัดจากรูป EXIF, location message, หรือข้อมูล GPS pipeline เพื่อเปรียบเทียบกับพิกัดลูกค้า และระบุได้ว่าช่างเข้าหน้างานจริงหรือไม่

#### 5.1.4 Voice-to-Workflow

ระบบนี้ช่วยให้ช่างสรุปงานด้วยเสียงแทนการพิมพ์ โดย workflow ที่ถูกต้องควรประกอบด้วยการรับไฟล์เสียง, ถอดเสียง, สรุปหัวข้อสำคัญ, ผูกเข้ากับ Job ID, และบันทึกผลเป็น structured log ลงฐานข้อมูล

| Voice message จากช่าง | Speech-to-Text → LLM summarization → mapping เข้าใบงาน | Raw text, summarized log, ประเด็นปัญหา, อะไหล่ที่ใช้ |

จากการตรวจสอบฐานข้อมูลปัจจุบัน แนะนำให้เพิ่มฟิลด์ `Voice_Summary_Log` ใน `DBJOBS` และเก็บ transcript/log ฉบับละเอียดในตารางแยกหากต้องการวิเคราะห์ย้อนหลัง

#### 5.1.5 Smart Tool Tags / แจ้งเตือนลืมเครื่องมือ

เมื่อช่างกดปิดงาน ระบบต้องไม่จบเพียงการอัปเดตสถานะ แต่ต้องต่อด้วยการแจ้งเตือนรายการเครื่องมือสำคัญที่เกี่ยวข้องกับงานนั้น เช่น สว่าน เครื่องวัดสายแลน บันได หรืองานติดตั้งเฉพาะทาง เพื่อป้องกันการลืมของไว้ที่หน้างาน

#### 5.1.6 Weekly Tool Audit

ระบบต้องส่งแจ้งเตือนทุกเย็นวันอาทิตย์ให้ช่างเช็คเครื่องมือผ่านหน้า LIFF หรือ checklist พร้อมแนบรูปยืนยัน การทำงานส่วนนี้มีบทบาทสำคัญต่อการควบคุมทรัพย์สินและวินัยการใช้งานเครื่องมือ และควรเก็บข้อมูลลง `DB_TOOL_AUDITS` โดยเฉพาะ

#### 5.1.7 AI Tech Companion

ฟีเจอร์นี้เป็นหนึ่งในส่วนขยายที่สำคัญของ V5.5 โดยมีเป้าหมายให้ช่างสามารถ “ถามบอทหน้างานได้เลย” ในสถานการณ์จริง เช่น ถามวิธีเข้าสาย, วิธีตั้งค่าอุปกรณ์, วิธีตรวจอาการเบื้องต้น, รายการอุปกรณ์ที่ควรใช้, หรือสถานะงานที่เกี่ยวข้องกับลูกค้ารายนั้น ระบบนี้ควรทำงานแบบ **context-aware assistant** ที่เชื่อมโยง 3 แหล่งข้อมูลสำคัญเข้าด้วยกัน ได้แก่

1. **Knowledge Base ของบริษัท** เช่น SOP, คู่มือติดตั้ง, มาตรฐานหน้างาน, FAQ
2. **ข้อมูลหน้างานจริง** เช่น Job ID, ประวัติลูกค้า, รูป Before/After, ปัญหาที่พบ, อะไหล่ที่เคยใช้
3. **ข้อมูลสต๊อกและการจัดซื้อ** เช่น สินค้าที่มีในรถช่างหรือคลังหลัก เพื่อช่วยตัดสินใจหน้างานได้จริง

ผู้ช่วยนี้ควรเป็นทั้ง **คู่มือหน้างาน**, **ตัวช่วยตัดสินใจ**, และ **ช่องทางค้นหาความรู้จากภายในองค์กร** โดยไม่บังคับให้ช่างออกจากแชทที่ใช้งานอยู่

#### 5.1.8 Smart Route

Smart Route เป็นฟีเจอร์ใหม่ที่ควรเชื่อมกับการมอบหมายงานและคิวงานของทีมช่าง โดยเน้นการใช้ข้อมูลพิกัดลูกค้า ตารางงาน และสภาพคิวงานเพื่อช่วยวางลำดับการเข้าหน้างานให้เหมาะสมขึ้น เป้าหมายไม่ใช่เพียงการนำทาง แต่คือการช่วยลดเวลาวิ่งรถ ลดการกลับร้านเพราะลืมของ และช่วยให้ dispatch เห็นภาระงานของแต่ละทีมได้ดีขึ้น

### 5.2 กลุ่มที่ 2: สต๊อก เครื่องมือ และการจัดซื้อ (Inventory, Asset & Procurement)

กลุ่มนี้มีหน้าที่ทำให้ธุรกิจ “รู้ของจริง” ในระดับที่นำไปใช้ตัดสินใจได้ ไม่ว่าจะเป็นสินค้าคงเหลือบนรถช่าง การโอนของจากคลังหลัก หรือการแจ้งเตือนก่อนเกิดปัญหาของขาด

#### 5.2.1 ระบบสต๊อก 3 ชั้น (3-Layer Inventory)

โมเดลสต๊อก 3 ชั้นเป็นพื้นฐานสำคัญของธุรกิจบริการติดตั้ง เพราะอุปกรณ์ไม่ได้อยู่ในที่เดียว ระบบต้องติดตามอย่างน้อย 3 ระดับ ได้แก่ **คลังหลัก**, **สต๊อกบนรถช่าง**, และ **เครื่องมือส่วนตัวช่าง** เพื่อให้ทราบว่าอุปกรณ์อยู่ที่ใคร อยู่คันไหน และถูกใช้ไปกับงานใด

จากการตรวจสอบฐานข้อมูลปัจจุบัน ตาราง `DB_INVENTORY` ยังไม่มีฟิลด์แยกประเภทตำแหน่งจัดเก็บ จึงควรเพิ่มอย่างน้อย `Location_Type` และ `Assigned_To` เพื่อให้รองรับโครงสร้างนี้

#### 5.2.2 Predictive Stocking

ระบบต้องวิเคราะห์คิวงานล่วงหน้าอย่างน้อย 3 วัน โดยดึง BOM หรือรายการอุปกรณ์ที่ต้องใช้มาเทียบกับ Van Stock ของแต่ละทีม หากของไม่พอ ระบบต้องแจ้งเตือนก่อนวันงาน ไม่ใช่รอให้ช่างพบปัญหาหน้างานแล้วจึงค่อยกลับไปเบิก

#### 5.2.3 Weekly Tool Audit

แม้เชิงการใช้งานจะอยู่ฝั่ง Field Operations แต่เชิงสินทรัพย์ถือเป็นส่วนสำคัญของ inventory control ด้วย เพราะช่วยสร้าง accountability ของช่างต่อเครื่องมือส่วนตัวและเครื่องมือที่เบิกใช้งานร่วมกัน

#### 5.2.4 Automated Draft PO

เมื่อสินค้าลดลงถึงจุดสั่งซื้อซ้ำ ระบบควรสร้างร่างใบสั่งซื้อ PDF ให้ทันที พร้อมข้อมูล vendor และรายการสินค้า เพื่อให้ผู้มีอำนาจอนุมัติได้จากแชทโดยไม่ต้องเริ่มเอกสารใหม่ทุกครั้ง ฟีเจอร์นี้เชื่อมกับ reorder point และ movement log โดยตรง

#### 5.2.5 Inventory Movement Intelligence

แม้ยังไม่ถูกตั้งชื่ออย่างเป็นทางการในเอกสารต้นทาง แต่จากโครงสร้างฐานข้อมูลปัจจุบันที่มี `DB_STOCK_MOVEMENTS` และ `DB_JOB_ITEMS` อยู่แล้ว ควรยกระดับให้ระบบสามารถอธิบายได้ว่า “ของชิ้นนี้ย้ายจากไหน ไปไหน ใช้กับงานใด ใครเป็นผู้ทำรายการ และคงเหลือหลังทำรายการเท่าไร” เพื่อรองรับทั้ง audit และการวิเคราะห์ต้นทุนจริงต่อใบงาน

### 5.3 กลุ่มที่ 3: การเงิน การขาย และเอกสารเชิงพาณิชย์ (Finance & Sales)

กลุ่มนี้มีเป้าหมายเพื่อให้การเสนอราคา รับชำระเงิน และออกเอกสารเกิดขึ้นอย่างรวดเร็วและตรวจสอบได้ โดยเฉพาะในบริบทธุรกิจที่ต้องทำงานหน้างานจริงและรับงานภาครัฐ

#### 5.3.1 PromptPay Automation

ทันทีที่งานใกล้ปิด ระบบต้องสามารถสร้าง QR พร้อมเพย์ที่มีจำนวนเงินตรงตามยอดคงค้างของใบงาน แล้วส่งให้ลูกค้าหรือให้ช่างเปิดแสดงหน้างานได้ทันที ช่วยลดความผิดพลาดจากการกรอกยอดเงิน และเร่งการเก็บเงินหลังจบงาน

#### 5.3.2 Slip Verification + Auto-Receipt

เมื่อลูกค้าส่งสลิป ระบบต้องอ่าน QR หรือ payload จากสลิป แล้วตรวจสอบยอดเงินและบัญชีผู้รับ หากตรงกับข้อมูลใบงานจึงออกใบเสร็จ PDF กลับให้อัตโนมัติ ฟีเจอร์นี้เป็นจุดตัดสำคัญระหว่างความเร็วและความปลอดภัยทางการเงิน เพราะช่วยลดความเสี่ยงสลิปปลอมและลดงานเอกสารของแอดมินการเงิน

จากฐานข้อมูลปัจจุบัน `DB_BILLING` ควรเพิ่มฟิลด์อย่างน้อยดังนี้

| ฟิลด์ใหม่ | วัตถุประสงค์ |
| --- | --- |
| `PromptPay_Payload` | เก็บข้อมูลที่ใช้สร้าง QR |
| `Slip_Verify_Status` | ระบุผลการตรวจสอบสลิป |
| `Slip_Image_ID` | ผูกหลักฐานสลิปกับบิล |
| `Receipt_PDF_URL` | เก็บลิงก์ใบเสร็จที่ออกแล้ว |

#### 5.3.3 Price Benchmarking

สำหรับงานภาครัฐหรือการจัดซื้อจัดจ้าง ระบบต้องสามารถเทียบราคากลาง ICT กับราคาที่กำลังใช้ในใบเสนอราคาได้ทันที ช่วยให้ฝ่ายขายไม่เสนอราคาเกินเงื่อนไข และลดเวลาในการตรวจเช็คราคาจากเอกสารภายนอก

#### 5.3.4 Automated Quotation

เซลส์หรือช่างควรสามารถเลือกแพ็กเกจสินค้าในหน้า LIFF แล้วให้ระบบสร้างใบเสนอราคา PDF ส่งให้ลูกค้าทันทีได้ ฟีเจอร์นี้สำคัญมากสำหรับงานปิดการขายหน้างาน และมีความสอดคล้องกับแนวคิดลดขั้นตอนการทำงานเอกสารซ้ำซ้อน

#### 5.3.5 Auto-TOR

Auto-TOR เป็นฟีเจอร์ใหม่ที่มีนัยสำคัญต่อธุรกิจของ COMPHONE เนื่องจากบริษัททำงานกับหน่วยงานราชการ ระบบควรถูกออกแบบให้สามารถช่วย **ร่างขอบเขตงานหรือรายการคุณลักษณะเฉพาะเบื้องต้น** จากข้อมูลสินค้า บริการ มาตรฐานราคา และข้อมูลหน้างานที่เก็บไว้แล้วในระบบ เป้าหมายคือช่วยลดเวลาการจัดทำเอกสารสำหรับโครงการราชการและทำให้เอกสารมีรูปแบบมาตรฐานมากขึ้น

ในระยะแรก ฟีเจอร์นี้ควรทำงานในระดับ **draft generator** ไม่ใช่การอนุมัติอัตโนมัติ กล่าวคือระบบช่วยรวบรวมรายการและเรียบเรียงฉบับร่างให้ผู้มีอำนาจตรวจทานก่อนทุกครั้ง

### 5.4 กลุ่มที่ 4: การตลาดและลูกค้าสัมพันธ์ (Marketing & CRM)

กลุ่มนี้เน้นการสร้างรายได้เพิ่มจากฐานข้อมูลที่ระบบเก็บไว้แล้ว โดยเฉพาะข้อมูลลูกค้าเดิม วันติดตั้ง พิกัด และรูปผลงาน

#### 5.4.1 AI Content Marketing

แม้ในเอกสารต้นทางจะระบุว่าฟีเจอร์นี้ถูกรวมเข้าใน Smart Photo Handling แล้ว แต่ในเชิง blueprint ควรถือเป็น capability สำคัญแยกต่างหาก เพราะมันเปลี่ยน “รูปหน้างาน” ให้กลายเป็น “สินทรัพย์ทางการตลาด” โดยอัตโนมัติ ตั้งแต่การคัดรูป การจัด layout การเขียนแคปชั่น ไปจนถึงการส่งร่างให้อนุมัติก่อนโพสต์

#### 5.4.2 Predictive Maintenance

ระบบควรใช้ข้อมูลวันติดตั้ง ประเภทลูกค้า และฤดูกาล เช่น ช่วงก่อนเข้าหน้าฝน หรือครบ 11 เดือนหลังติดตั้ง เพื่อส่งข้อความชวนตรวจเช็กหรือเสนอแพ็กเกจบำรุงรักษา เป็นกลไกสร้างรายได้ซ้ำจากลูกค้าเดิมที่ใช้ต้นทุนการได้มาน้อยกว่าลูกค้าใหม่

#### 5.4.3 Heatmap Analytics

ฟีเจอร์นี้ช่วยให้ผู้บริหารมองเห็นความหนาแน่นของลูกค้าในพื้นที่ต่าง ๆ ผ่านแผนที่ความร้อน โดยใช้พิกัดจาก `DB_CUSTOMERS` เป็นฐาน เหมาะสำหรับการตัดสินใจเรื่องพื้นที่ยิงโฆษณา การวางแผนเปิดหน่วยบริการ หรือการจัดทีมงานตามภูมิศาสตร์

#### 5.4.4 Auto-CSAT

Auto-CSAT เป็นฟีเจอร์ใหม่ที่ควรถูกเพิ่มหลังจบงานหรือหลังปิดการเงิน โดยระบบจะส่งแบบประเมินความพึงพอใจแบบสั้นให้ลูกค้าผ่าน LINE เพื่อเก็บคะแนน รีวิว และ feedback เชิงข้อความ ข้อมูลนี้สามารถนำไปใช้ได้ 3 ทางพร้อมกัน คือ

1. ใช้วัดคุณภาพทีมช่างและคุณภาพงานจริง
2. ใช้กระตุ้นการ follow-up ลูกค้าที่ไม่พึงพอใจ
3. ใช้คัดเลือกรูปงานและรีวิวที่ดีไปใช้ต่อในงานการตลาด

#### 5.4.5 Customer Self-Service และ IoT Integration ในฐานะฟีเจอร์ต่อยอด

จากเอกสารข้อเสนอเพิ่มเติม ฟีเจอร์ที่ควรพิจารณาในระยะถัดไป ได้แก่ พอร์ทัลให้ลูกค้าแจ้งซ่อมด้วยตนเอง, การเชื่อมต่อ monitoring สำหรับอุปกรณ์องค์กร, ระบบ gamification ของช่าง, และ e-signature สำหรับเอกสารรับมอบงาน ซึ่งทั้งหมดมีความเหมาะสมในฐานะ roadmap หลังจากฟังก์ชันหลัก V5.5 เสถียรแล้ว

## 6. Cross-Room Automation Flow: 1 งาน เชื่อม 4 ห้อง

หนึ่งในจุดเด่นที่สุดของโครงการคือการออกแบบ flow ที่เชื่อม **Sales → Field Tech → Finance → Procurement** เข้าด้วยกันโดยให้บอทเป็นตัวส่งต่องานอัตโนมัติ นี่คือกลไกที่ช่วยลด “งานตกหล่น” และ “การตามงาน” ซึ่งเป็น pain point หลักของธุรกิจบริการภาคสนาม

| ห้อง / แผนก | เหตุการณ์หลัก | สิ่งที่ระบบต้องทำ |
| --- | --- | --- |
| **ห้องการขาย** | เปิดงานใหม่และระบุรายละเอียดลูกค้า | สร้าง Job ID, สร้างโฟลเดอร์ Drive, ส่งต่อข้อมูลไปห้องช่าง |
| **ห้องช่าง** | สแกน QR, ส่งรูป, อัปเดตสถานะ, ใช้อุปกรณ์ | จัดเก็บรูป, วิเคราะห์ AI, อัปเดตสถานะ, ตัดสต๊อก, แจ้งห้องการเงิน |
| **ห้องการเงิน** | ออก invoice, รับสลิป, ตรวจสอบการชำระ | ตรวจสลิป, ออกใบเสร็จ, อัปเดตสถานะการเงิน, แจ้งห้องผู้บริหาร |
| **ห้องสั่งสินค้า** | พบของต่ำกว่าจุดสั่งซื้อ | แจ้งเตือน, สร้าง draft PO, ส่งให้อนุมัติ |

คุณค่าของ flow นี้ไม่ได้อยู่ที่การส่งข้อความระหว่างห้องเพียงอย่างเดียว แต่คือการทำให้ทุกเหตุการณ์มี **data trace**, **ownership**, และ **trigger logic** ที่ต่อเนื่องกันทั้งระบบ

| `INVENTORY`, `DBATTENDANCE` | ชีตเก่า/ซ้ำซ้อน | ควรวางแผนมาตรฐานชื่อใหม่ |

## 10. Archive + Performance Strategy

จากขนาดและลักษณะการใช้งานของ Google Apps Script + Google Sheets จำเป็นต้องมีแนวทางจัดการข้อมูลและประสิทธิภาพตั้งแต่ต้น ไม่เช่นนั้นระบบจะช้าลงเมื่อข้อมูลโตขึ้น โดยเฉพาะตารางใบงาน รูปภาพ และ log

### 10.1 กลยุทธ์ Archive

| ประเภทข้อมูล | ระยะจัดเก็บแบบ Active | แนวทาง archive |
| --- | --- | --- |
| **DBJOBS และ DB_JOB_LOGS** | งานที่ยังเปิดอยู่ + งานย้อนหลัง 12 เดือน | ย้ายงานปิดแล้วไป workbook archive รายปี |
| **DB_PHOTO_QUEUE** | เก็บแถวสรุปสำหรับงาน active และงานล่าสุด | เก็บไฟล์จริงบน Drive ต่อ แต่ย้าย metadata เก่าไป archive sheet |
| **DB_SYSTEM_LOGS** | เก็บ log ปฏิบัติการล่าสุดที่จำเป็นต่อ monitoring | archive รายเดือน/ไตรมาส |
| **DB_BILLING** | เก็บข้อมูล active สำหรับปีปัจจุบัน | archive ตามรอบบัญชีหรือปีภาษี |

### 10.2 กลยุทธ์ Performance

1. **ใช้ batch read/write** แทนการอ่านเขียนทีละเซลล์
2. **แยก hot tables ออกจาก historical tables** โดยเฉพาะใบงานและ system logs
3. **ใช้ queue processing กับงานหนัก** เช่น การประมวลผลรูปภาพหรือไฟล์เสียงจำนวนมาก
4. **ทำ caching สำหรับข้อมูล lookup** เช่น รายชื่อช่าง รายชื่อลูกค้า และรายการสินค้า
5. **ออกแบบ idempotent workflow** เพื่อป้องกันการรันซ้ำแล้วเขียนข้อมูลซ้ำ
6. **แบ่งงานระยะยาวเป็น chunk** ให้เหมาะกับข้อจำกัดเวลารันของ GAS
7. **กำหนด retention policy ของรูปและไฟล์** โดยแยกไฟล์หลักฐานที่ต้องเก็บนานออกจากไฟล์ชั่วคราว

แนวทางนี้สอดคล้องกับข้อจำกัดจริงของ Apps Script ทั้งเรื่อง execution time, concurrent executions, และ quota ของ external API

## 12. ความเสี่ยงและข้อจำกัดทางเทคนิค

| ประเด็น | ความเสี่ยง | แนวทางรับมือ |
| --- | --- | --- |
| **Execution time ของ GAS** | งานรูปและไฟล์เสียงจำนวนมากอาจ time-out | ใช้ queue, chunking, worker pattern |
| **API quota และ email limits** | ใช้งานหนักอาจชน quota | monitoring, caching, retry policy |
| **Concurrent executions** | ช่างส่งข้อมูลพร้อมกันอาจเกิดคอขวด | backoff, lock, queue sequencing |
| **ความแม่นยำ GPS/EXIF** | บางอุปกรณ์อาจไม่มี location | ใช้ location message ควบคู่ |
| **ต้นทุน AI/API ภายนอก** | ค่าใช้จ่ายเพิ่มตามการใช้งาน | ใช้เฉพาะจุดที่คุ้มค่า, กำหนด rate limit |
| **Schema drift ใน Google Sheets** | ตารางซ้ำซ้อนและชื่อไม่มาตรฐาน | data governance และ migration plan |

## 14. บทสรุปเชิงยุทธศาสตร์

COMPHONE SUPER APP V5.5 ไม่ใช่เพียงการเพิ่มฟีเจอร์ใหม่ให้ระบบเดิม แต่เป็นการปรับวิธีทำงานขององค์กรให้กลายเป็น **ระบบปฏิบัติการดิจิทัลสำหรับธุรกิจไอทีและงานติดตั้ง** ที่ใช้ข้อมูลจริงเป็นศูนย์กลาง ใช้ AI ในงานที่คุ้มค่า และใช้แชทเป็น interface หลักในการขับเคลื่อนงาน

จุดแข็งสำคัญของโครงการนี้คือการมีฐานระบบเดิมที่ใช้งานได้แล้วหลายส่วน เช่น dual-engine messaging, photo queue, vision analysis, inventory core, document generation และฐานข้อมูล Google Sheets ที่มีโครงสร้างใช้งานจริง เมื่อรวมกับการเพิ่ม state machine, voice workflow, QR รับเงิน, slip verification, AI Tech Companion, Smart Route, Auto-TOR และ Auto-CSAT ระบบจะก้าวจาก “เครื่องมือช่วยงาน” ไปสู่ “แพลตฟอร์มปฏิบัติการของทั้งบริษัท” ได้อย่างแท้จริง

หากดำเนินการตาม blueprint นี้อย่างมีลำดับ โดยเริ่มจากโครงสร้างข้อมูลและ flow หลักที่สร้างผลกระทบสูงก่อน V5.5 จะสามารถสร้างคุณค่าได้ทั้งในมิติ **ความเร็ว**, **ความถูกต้อง**, **ความโปร่งใส**, **การควบคุมต้นทุน**, และ **การเติบโตของรายได้จากฐานลูกค้าเดิม** พร้อมกัน
